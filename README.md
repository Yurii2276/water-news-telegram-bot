# Water News Intelligent Editor

Інтелектуальний редактор україномовних новин про водопостачання та
водовідведення. Система збирає матеріали, перевіряє тематику, усуває дублікати,
класифікує зміст через AI та надсилає адміну редакційні картки.

## Можливості

- офіційні джерела: НКРЕКП, Кабмін, Верховна Рада, профільні комітети,
  Мінрозвитку, Держводагентство та Асоціація міст України;
- Google News використовується лише для discovery; нерозв'язаний URL Google
  ніколи не потрапляє до редакційної черги;
- тематичний pre-filter, AI-класифікація зі strict JSON Schema та дедуплікація;
- PostgreSQL як журнал матеріалів і статусів;
- автоматична публікація лише матеріалів із релевантністю та довірою не нижче
  85%;
- ліміт 10 публікацій на добу, інтервал, повторні спроби й технічний звіт;
- ручний `/scan` і щоденне автоматичне сканування.

## Запуск

Потрібен Node.js 22 або новіший.

```powershell
Copy-Item .env.example .env
```

Створіть бота через [@BotFather](https://t.me/BotFather). Для локального запуску
скопіюйте `.env.example` у `.env` та заповніть секрети. `.env` не комітьте.

```powershell
npm install
npm start
```

У цьому workspace також встановлено portable Node.js:

```powershell
$env:Path = "$(Resolve-Path .\.tools\node-v24.18.0-win-x64);$env:Path"
.\.tools\node-v24.18.0-win-x64\npm.cmd install
.\.tools\node-v24.18.0-win-x64\npm.cmd start
```

## Редакційний процес

1. Collector читає офіційні сторінки та Google News RSS.
2. Pre-filter вимагає прямий контекст водопостачання/водовідведення; одного
   слова «відновлення», «закон» чи «форум» недостатньо.
3. Для Google-кандидата витягується URL першоджерела. Якщо він не визначений,
   кандидат отримує статус `filtered_out`.
4. Матеріал дедуплікується за нормалізованим URL, заголовком і Jaccard-подібністю
   змісту.
5. AI бачить лише текст першоджерела. Для публікації потрібні
   `relevanceScore >= 85` і `confidenceScore >= 85`.
6. Перед відправленням URL повторно перевіряється: він має відкриватися та
   залишатися в реєстрі надійних джерел.
7. Матеріал автоматично публікується з урахуванням добового ліміту й інтервалу.
   Невдалі спроби журналюються; після вичерпання retry матеріал відхиляється.

Статуси: `filtered_out`, `duplicate`, `rejected_source`, `rejected_ai`,
`queued`, `dry_run`, `published`, `rejected_publish`.

## Безпечний DRY_RUN

`DRY_RUN=true` — типове й безпечне значення. Pipeline виконує discovery,
фільтрацію, дедуплікацію, AI та перевірку URL, але не викликає надсилання в
Telegram-канал. Перевірені матеріали записуються зі статусом `dry_run`.

Для реального запуску явно встановіть `DRY_RUN=false`. Під час старту матеріали,
перевірені у dry-run, автоматично повернуться до черги. Не перемикайте режим,
доки канал і всі Railway Variables не перевірені.

## Команди адміністратора

- `/scan` — запустити сканування;
- `/queue` — показати чергу автопублікації;
- `/news` — показати останні опубліковані матеріали;

## Підготовка до Railway

Проєкт налаштований як long-polling worker через `railway.json`. HTTP-сервер,
публічний домен і змінна `PORT` йому не потрібні.

Коли будете готові до деплою:

1. Підключіть GitHub-репозиторій до нового Railway service.
2. У вкладці **Variables** додайте обов'язкові змінні з `.env.example`.
3. Додайте PostgreSQL service і передайте його `DATABASE_URL`.
4. Залиште одну replica: Telegram long polling не повинен одночасно працювати
   з кількох екземплярів одного бота.

Файл `.env` не комітьте: секрети мають зберігатися у Railway Variables.
